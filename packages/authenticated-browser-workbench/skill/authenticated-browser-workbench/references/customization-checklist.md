# 个性化改造清单

## 1. 先定义任务边界

- 哪些问题只用AI默认网页搜索就足够？
- 出现哪些信息缺口时应自动升级到登录态检索？
- 不同类型的缺口分别优先进入哪个网站，最多补充几个渠道？
- 哪些任务必须复用登录态？
- 哪些公开URL应继续走公开网页读取？
- 哪些目标是原生App或已有专用连接器？
- 哪些动作可以只读执行，哪些动作必须逐次确认？

## 2. 梳理浏览器和账号

- 为每个常用浏览器写明主要用途，不用“默认浏览器”代替任务路由。
- 标出本人、家人、单位和测试账号的边界。
- 把禁止使用的组合写成硬规则，例如“浏览器A中的服务B不得使用”。
- 不记录凭据或本机Profile路径。

## 3. 只做代表性测试

按以下顺序收口，不追求穷尽所有网站：

1. 单浏览器连接与会话停止；
2. 登录态页面读取；
3. 一次只读站内搜索；
4. 多浏览器实例显式选择；
5. 浏览器完整重启后的复用；
6. 一个需要人工接管或失败回退的场景。

每个非核心失败最多执行一次修复和一次复测。连接、账号边界、实例选择或会话停止失败时阻断发布；单个站点受限时写入限制，不无限测试。

## 4. 把经验写成规则

- 记录“公开搜索缺少什么”和“登录态渠道补足了什么”，不把调用浏览器本身当作成功。
- 记录稳定的选择原则，不记录一次性的元素编号。
- 记录“哪个阶段没有完成”，不笼统写“不可用”。
- 记录替代渠道和停止条件，避免下一次重复死循环。
- 当前事实替换旧结论；历史过程放在项目测试记录，不堆进运行Skill。

## 5. 分离个人版和公开版

个人版可以保存浏览器用途、账号禁区和私有站点矩阵；公开版只发布可填写模板、通用方法、匿名测试环境、隐私说明和已知限制。发布前搜索并移除：

- 姓名、邮箱、手机号和账号关系；
- 本机绝对路径、实例ID和会话ID；
- 历史对话、搜索词、收藏、草稿、消息和后台数字；
- Cookie、Token、API Key、验证码和恢复码；
- 未经许可的第三方代码、图标和品牌资产。

## 6. 公开候选与正式版

公开候选表示核心方法已经可交付，但环境覆盖和长期稳定性仍有限。正式版应在候选包经过独立安装、真实调用、隐私复核和一段实际使用后再决定，不因页面已经公开就自动升级状态。
